## 1. ISTQB / ISO 29119 软件测试标准

### 1.1 核心框架

| 标准 | 说明 |
|------|------|
| **ISTQB** | International Software Testing Qualifications Board，业界最广泛采用的测试认证体系 |
| **ISO 29119** | ISO/IEEE 29119 软件测试标准族，包含 5 个部分，定义了测试过程、文档、测试设计技术等 |

### 1.2 测试级别 (Test Levels)

```
┌─────────────────────────────────────────────────────────┐
│                    验收测试 (UAT)                        │
│         用户验收，验证系统是否满足业务需求                 │
├─────────────────────────────────────────────────────────┤
│                    系统测试 (SIT)                        │
│       端到端验证系统功能、性能、安全性等完整功能           │
├─────────────────────────────────────────────────────────┤
│                    集成测试 (IT)                         │
│        验证模块/服务之间的接口和数据交互是否正确           │
├─────────────────────────────────────────────────────────┤
│                    单元测试 (UT)                         │
│          最小代码单元的验证，通常是函数或方法             │
└─────────────────────────────────────────────────────────┘
```

**关键指标**：

- 单元测试覆盖率：核心业务代码 ≥ 80%
- 集成测试：每个接口/模块至少 1 个正向 + 1 个负向用例
- 系统测试：所有功能需求必须有对应的测试用例
- 验收测试：用户故事覆盖率 100%

### 1.3 测试类型 (Test Types)

| 类型 | 目标 | 适用级别 |
|------|------|----------|
| **功能测试** | 验证功能正确性 | UT/IT/SIT/UAT |
| **性能测试** | 响应时间、吞吐量、并发 | SIT/UAT |
| **安全测试** | 漏洞、权限、加密 | SIT/UAT |
| **回归测试** | 确保变更未破坏已有功能 | 全级别 |
| **冒烟测试** | 核心流程快速验证 | SIT |

## 2. Google Software Testing Philosophy

### 2.1 核心原则

Google 将测试分为三类：

1. **小型测试 (Small Tests)**：单元测试，验证单一函数行为，无 I/O，毫秒级执行
2. **中型测试 (Medium Tests)**：集成测试，验证组件间交互，可有轻量 I/O
3. **大型测试 (Large Tests)**：E2E 测试，模拟真实用户操作，完整环境

### 2.2 规模化测试实践

```
测试金字塔（Google 推荐）:
         ▲
        /E\        大型测试 (10-15%)
       /EEEE\
      /IIIIII\     中型测试 (20-30%)
     /UUUUUUUUU\   小型测试 (60-70%)
    /UUUUUUUUUUUUU\
─────────────────────────────
```

**Google 的关键经验**：
- 测试速度决定开发速度：小型测试必须在毫秒级完成
- 测试可靠性优先：flaky test 危害大于没有测试
- 隔离依赖：使用 mock/stub 避免外部系统依赖
- 测试代码质量与产品代码同等重要

### 2.3 内网环境适配

对于 ITOps 平台这类内网系统：
- 减少对外部网络的依赖，使用 mock 模拟外部服务
- 大型测试可在隔离的测试环境中运行
- 中型测试覆盖核心模块间交互

## 3. 测试金字塔

### 3.1 经典比例

| 层级 | 比例 | 执行时间 | 覆盖范围 |
|------|------|----------|----------|
| **E2E (Large)** | 10-15% | 分钟级 | 关键用户路径 |
| **集成 (Medium)** | 20-30% | 秒级 | 模块接口 |
| **单元 (Small)** | 60-70% | 毫秒级 | 核心逻辑 |

### 3.2 针对 ITOps 平台的调整建议

```
推荐比例（内网高可靠性系统）:
       /M\           15% 中型集成测试
      /MMMM\
     /UUUUUU\         55% 单元测试（核心模块）
    /UUUUUUUU\
   /IIIIIIIIIII\      30% 集成测试（模块间）
  /IIIIIIIIIIIIII\
 /LLLLLLLLLLLLLLLLLL\ 少量 E2E（关键场景）
```

**理由**：
- 内网系统强调稳定性和快速反馈
- 单元测试比例可略低，但核心模块应达到 80%+
- 集成测试比例提高，确保模块间协作可靠
- E2E 只覆盖最关键的运维场景

## 4. TDD / BDD 最佳实践

### 4.1 TDD (Test-Driven Development) 流程

```
红(Red) → 绿(Green) → 重构(Refactor)
 │         │           │
 ▼         ▼           ▼
写一个会  让它通过    优化代码
失败的测试             
```

**实施步骤**：
1. **Red**: 编写一个描述期望行为的测试（快速失败）
2. **Green**: 编写最少量代码让测试通过（不求完美）
3. **Refactor**: 重构代码，消除重复，提升质量
4. 重复循环

### 4.2 BDD (Behavior-Driven Development)

BDD 是 TDD 的演进，强调用自然语言描述行为：

```
功能: 用户登录
  场景: 正确凭据登录成功
    假如 用户在登录页面
    当 用户输入正确的用户名和密码
    那么 系统显示用户仪表板

  场景: 错误凭据登录失败
    假如 用户在登录页面
    当 用户输入错误的密码
    那么 系统显示错误提示
```

**常用工具**：
- Java/Go: Ginkgo, GoConvey
- Python: Behave, pytest-bdd

### 4.3 ITOps 平台实施建议

- **核心模块（配置管理、任务调度）**：采用 TDD，测试先行
- **业务工作流（告警处理、工单流转）**：采用 BDD，用自然语言描述场景
- **工具类函数**：纯 TDD，边界值驱动

## 5. API 集成测试最佳实践

### 5.1 REST API 测试策略

**测试分层**：

```
┌──────────────────────────────────────┐
│         Contract Tests (可选)         │  验证 API 契约
├──────────────────────────────────────┤
│         Integration Tests             │  端到端 API 测试
├──────────────────────────────────────┤
│           Mock/Stub                  │  隔离外部依赖
└──────────────────────────────────────┘
```

### 5.2 测试用例设计

**针对每个 API 端点，至少覆盖**：

| 测试类别 | 示例 |
|----------|------|
| **正向** | `GET /api/v1/servers` → 200 + 正确数据 |
| **负向** | `GET /api/v1/servers/99999` → 404 |
| **边界** | `GET /api/v1/servers?page=0&size=1000` |
| **权限** | 未认证请求 → 401 |
| **参数校验** | 缺少必填字段 → 400 |

### 5.3 关键指标

- 每个 API 端点至少 3 个测试用例（正向/负向/异常）
- 测试覆盖率 ≥ 80% 的 API 逻辑分支
- 平均响应时间 SLA 监控

## 6. 混沌工程 / 金丝雀发布（可选高级实践）

### 6.1 混沌工程

适用于 ITOps 平台这类高可靠性系统：

**核心原则**：
- 在非生产环境验证系统在故障下的行为
- 模拟：网络中断、服务宕机、磁盘满、CPU 飙升
- 观察：系统恢复时间、告警触发、数据完整性

**工具**：Chaos Monkey（Netflix）、LitmusChaos（K8s）

### 6.2 金丝雀发布

```
[旧版本] ───→ [金丝雀(10%)] ───→ [新版本(100%)]
             ↓                    ↓
          监控指标             监控指标
          错误率               错误率
```

**关键指标**：
- 流量切换比例：10% → 30% → 100%
- 监控窗口：每阶段至少 15-30 分钟
- 自动回滚阈值：错误率 > 1%

### 6.3 内网环境建议

对于内网 ITOps 平台：
- **混沌工程**：可在测试环境选择性实施，重点验证告警链路
- **金丝雀发布**：推荐使用，适用于配置变更和版本升级
- 优先保证：变更可回滚、监控可观测

## 7. 代码覆盖率标准

### 7.1 行业推荐目标

| 系统类型 | 行覆盖率 | 分支覆盖率 | 备注 |
|----------|----------|------------|------|
| 通用业务系统 | 70-80% | 60-70% | 最低标准 |
| 核心/关键模块 | 80-90% | 70-80% | 推荐目标 |
| 高可靠性系统 | 90%+ | 80%+ | 严格标准 |
| 测试充分性 | MCDC 100% | - | 认证标准 |

### 7.2 覆盖率指标定义

- **行覆盖率 (Line Coverage)**：代码中被执行到的行数 / 总行数
- **分支覆盖率 (Branch Coverage)**：每个判断分支是否都被执行
- **MC/DC (Modified Condition/Decision Coverage)**：航电/汽车等安全关键系统要求

### 7.3 ITOps 平台覆盖率目标

```
目标覆盖率（分模块）：
┌─────────────────────┬──────────┬──────────┐
│       模块          │ 行覆盖   │ 分支覆盖 │
├─────────────────────┼──────────┼──────────┤
│ 核心业务逻辑         │   ≥80%   │   ≥70%   │
│ API/接口层          │   ≥70%   │   ≥60%   │
│ 工具/辅助函数        │   ≥60%   │   ≥50%   │
│ 配置/参数处理        │   ≥90%   │   ≥80%   │
└─────────────────────┴──────────┴──────────┘
```

**注意**：覆盖率是手段而非目标，测试有效性（能否发现缺陷）比覆盖率更重要。

## 8. 测试用例设计原则

### 8.1 边界值分析 (Boundary Value Analysis)

**原则**：缺陷往往出现在边界条件

```
示例：服务器列表分页
- 最小页: page=1
- 最后一个有效页: page=last
- 边界: page=0, page=last+1
- 典型: page=1, page=100

对于范围 [min, max]:
测试: min-1, min, min+1, typical, max-1, max, max+1
```

### 8.2 等价类划分 (Equivalence Partitioning)

**原则**：具有相同行为的输入归为一类，抽样测试

```
示例：CPU 使用率告警阈值
- 有效等价类: 0-70%（正常）
- 有效等价类: 71-90%（警告）
- 无效等价类: < 0
- 无效等价类: > 100

每个等价类只需测试一个代表值
```

### 8.3 场景法 (Scenario Testing)

**原则**：用业务流程串联多个功能点

```
告警处理完整场景:
1. 触发告警（监控指标超阈值）
2. 告警生成（状态: New）
3. 值班人员认领（状态: Acknowledged）
4. 问题定位（状态: Investigating）
5. 问题解决（状态: Resolved）
6. 告警关闭（状态: Closed）

每个状态转换都是测试点
```

### 8.4 组合测试 (Pairwise Testing)

**原则**：使用正交表，用最少的用例覆盖最多的参数组合

```
场景：服务器健康检查
- 操作系统: [Linux, Windows]
- 内存状态: [正常, 不足]
- 磁盘状态: [正常, 满]
- 网络状态: [正常, 中断]

全组合: 2×2×2×2 = 16 种
Pairwise: 只需 6-8 种即可覆盖所有两两组合
```

## 9. ITOps 平台测试策略建议

### 9.1 分层测试策略

```
┌─────────────────────────────────────────────────┐
│  验收测试 (UAT)                                  │
│  - 运维场景端到端验证                             │
│  - 关键业务流（告警→处理→复盘）                   │
├─────────────────────────────────────────────────┤
│  系统测试 (SIT)                                  │
│  - REST API 全覆盖                              │
│  - 性能基准测试（响应时间 < 200ms）               │
│  - 安全扫描（OWASP Top 10）                      │
├─────────────────────────────────────────────────┤
│  集成测试 (IT)                                   │
│  - 模块间接口测试                                │
│  - 数据库读写验证                                │
│  - 消息队列/缓存交互                             │
├─────────────────────────────────────────────────┤
│  单元测试 (UT)                                   │
│  - 核心算法和业务逻辑                            │
│  - 工具函数                                      │
│  - 配置解析和验证                                │
└─────────────────────────────────────────────────┘
```

### 9.2 自动化测试目标

| 级别 | 自动化率目标 | 执行频率 |
|------|-------------|----------|
| 单元测试 | 95%+ | 每次提交 |
| 集成测试 | 80%+ | 每日构建 |
| API 测试 | 90%+ | 每次部署 |
| E2E 测试 | 50%+ | 每周回归 |

### 9.3 内网环境特殊考虑

- **离线环境**：所有测试数据本地生成，避免外部依赖
- **安全隔离**：测试环境与生产环境网络隔离
- **数据模拟**：使用 mock 对象模拟数据库和中间件
- **日志审计**：测试执行日志完整保留，便于问题追溯
